# Phase 4 Context: Integration Infrastructure

## 锁定的实现决策

### 技能发现机制
- **决策**：使用 Glob 工具自动发现 SKILL.md 文件
- **理由**：Glob 工具是 iFlow CLI 提供的标准工具，能够高效地查找文件
- **实现**：
  - 使用模式 `skills/*/SKILL.md` 查找所有技能
  - 使用模式 `skills/*/examples/README.md` 查找所有示例
  - 使用模式 `skills/*/tests/test-basic.ps1` 查找所有测试
  - 解析 YAML frontmatter 提取元数据

### 工具映射系统
- **决策**：创建统一的工具映射配置文件
- **理由**：集中管理工具映射，便于维护和更新
- **实现**：
  - 创建 `tool-mapping.json` 配置文件
  - 定义 Superpowers 工具到 iFlow CLI 工具的映射
  - 提供工具映射的验证和测试

### 子代理系统适配
- **决策**：创建子代理类型映射配置
- **理由**：iFlow CLI 不支持自定义代理定义，需要将 Superpowers 的自定义代理映射到预定义类型
- **实现**：
  - 创建 `subagent-mapping.json` 配置文件
  - 定义 Superpowers 代理类型到 iFlow CLI 代理类型的映射
  - 在 prompt 中明确指定代理角色

### 技能加载机制
- **决策**：使用 read_file 工具动态加载技能内容
- **理由**：read_file 工具是 iFlow CLI 提供的标准工具，能够读取文件内容
- **实现**：
  - 根据技能名称定位 SKILL.md 文件
  - 读取文件内容并解析 YAML frontmatter
  - 提取技能元数据和内容

### 钩子系统
- **决策**：创建钩子配置文件
- **理由**：钩子系统可以在特定事件触发时注入技能上下文
- **实现**：
  - 创建 `hooks.json` 配置文件
  - 定义 SessionStart 钩子
  - 在钩子中加载并注入技能上下文

## Phase 4 的限制

- **时间限制**：每个机制的实现时间不超过 2 小时
- **复杂度限制**：每个机制的实现不超过 500 行代码
- **依赖限制**：不依赖外部库或工具
- **测试限制**：每个机制至少有 2 个测试用例
- **文档限制**：每个机制必须有清晰的使用说明
- **兼容性限制**：必须与 iFlow CLI v0.4.11+ 兼容

## Phase 4 的成功标准

1. 技能可以自动发现和加载
2. Superpowers 工具正确映射到 iFlow CLI 工具
3. 子代理系统正常工作
4. 钩子系统可以注入技能上下文
5. 每个机制至少有 2 个测试用例
6. 每个机制有清晰的使用说明
7. 工具映射表完整且准确
8. 子代理映射策略明确
9. 系统性能满足要求
10. 与 iFlow CLI 完全兼容
