# Subagent-Driven Development

## 概述

通过为每个任务分发新的子代理并执行两阶段审查来执行计划：首先是规范合规性，然后是代码质量。

**核心原则**: 每个任务使用新的子代理 + 两阶段审查（规范 → 质量）= 高质量、快速迭代

**开始时声明**: "我正在使用 Subagent-Driven Development 来执行此计划。"

## 何时使用

使用 Subagent-Driven Development 当：

- 您有实施计划
- 任务主要是独立的
- 您想保持在当前会话中
- 您需要快速迭代和自动审查检查点

**vs. 执行计划（并行会话）**:
- 相同的会话（无上下文切换）
- 每个任务使用新的子代理（无上下文污染）
- 每个任务后两阶段审查：规范合规性，然后代码质量
- 更快的迭代（任务之间无人工循环）

## 流程

### 1. 读取计划并提取任务

读取计划文件一次，提取所有任务及其完整文本，记录上下文，创建 TodoWrite。

**工具映射**:
- `Read` → `read_file` - 用于读取计划文件
- `TodoWrite` → `todo_write` - 用于创建待办事项列表

### 2. 对于每个任务：

对于每个任务执行以下步骤：

#### a. 分发实现者子代理

使用 `task` 工具分发 `general-purpose` 子代理，提供完整的任务文本和上下文。

**Prompt**: 见 `prompts/implementer-prompt.md`

**工具映射**:
- `Task` → `task` - 用于分发子代理，指定 `subagent_type: "general-purpose"`

#### b. 处理子代理问题

如果子代理提出问题：
- 清晰完整地回答
- 提供额外上下文（如果需要）
- 不要催促他们进入实现

#### c. 子代理实现、测试、提交、自我审查

实现者子代理：
- 实现任务
- 编写/运行测试
- 提交代码
- 自我审查

#### d. 分发规范审查者子代理

使用 `task` 工具分发 `plan-agent` 子代理进行规范合规性审查。

**Prompt**: 见 `prompts/spec-reviewer-prompt.md`

**工具映射**:
- `Task` → `task` - 用于分发子代理，指定 `subagent_type: "plan-agent"`

#### e. 规范审查

如果规范审查者发现问题：
- 实现者子代理（同一个子代理）修复问题
- 规范审查者再次审查
- 重复直到批准

#### f. 分发代码质量审查者子代理

使用 `task` 工具分发 `general-purpose` 子代理进行代码质量审查。

**Prompt**: 见 `prompts/code-quality-reviewer-prompt.md`

**工具映射**:
- `Task` → `task` - 用于分发子代理，指定 `subagent_type: "general-purpose"`

#### g. 代码质量审查

如果代码质量审查者发现问题：
- 实现者子代理（同一个子代理）修复问题
- 代码质量审查者再次审查
- 重复直到批准

#### h. 标记任务完成

在 TodoWrite 中标记任务为 completed。

**工具映射**:
- `TodoWrite` → `todo_write` - 用于更新任务状态

### 3. 更多任务？

检查 TodoWrite，如果更多任务存在，返回到步骤 2。

**工具映射**:
- `TodoRead` → `todo_read` - 用于读取任务状态

### 4. 最终代码审查

分发最终的代码审查者子代理进行整个实现的审查。

**工具映射**:
- `Task` → `task` - 用于分发子代理，指定 `subagent_type: "general-purpose"`

### 5. 完成开发

声明："我正在使用 finishing-a-development-branch 技能来完成这项工作。"

**必需子技能**：使用 `Skill` 工具调用 finishing-a-development-branch

**工具映射**:
- `Skill` → `Skill` - 用于调用 finishing-a-development-branch 子技能

## Prompt 模板

- `prompts/implementer-prompt.md` - 分发实现者子代理
- `prompts/spec-reviewer-prompt.md` - 分发规范合规性审查者子代理
- `prompts/code-quality-reviewer-prompt.md` - 分发代码质量审查者子代理

## 优势

**vs. 手动执行**:
- 子代理自然遵循 TDD
- 每个任务的全新上下文（无混淆）
- 并行安全（子代理不干扰）
- 子代理可以在之前和期间提出问题

**vs. 执行计划**:
- 相同的会话（无移交）
- 持续进度（无等待）
- 审查检查点自动

**效率收益**:
- 无文件读取开销（控制器提供完整文本）
- 控制器精确策划所需上下文
- 子代理预先获得完整信息
- 在工作开始之前提出问题（而不是之后）

**质量门禁**:
- 自我审查在移交前捕获问题
- 两阶段审查：规范合规性，然后代码质量
- 审查循环确保修复实际工作
- 规范合规性防止过度/不足构建
- 代码质量确保实现良好构建

**成本**:
- 更多子代理调用（每个任务实现者 + 2 个审查者）
- 控制器做更多准备工作（提前提取所有任务）
- 审查循环增加迭代
- 但早期捕获问题（比以后调试更便宜）

## 红色警告

**从不**:
- 在没有明确用户同意的情况下在 main/master 分支上开始实现
- 跳过审查（规范合规性或代码质量）
- 用未修复的问题继续
- 并行分发多个实现子代理（冲突）
- 让子代理读取计划文件（提供完整文本代替）
- 跳过场景设置上下文（子代理需要理解任务在哪里适应）
- 忽略子代理问题（在让他们继续之前回答）
- 接受规范合规性的"足够好"（规范审查者发现问题 = 未完成）
- 跳过审查循环（审查者发现问题 = 实现者修复 = 再次审查）
- 让实现者自我审查替换实际审查（两者都需要）
- **在规范合规性是 ✅ 之前开始代码质量审查**（错误顺序）
- 当任一审查有未解决问题时移动到下一个任务

**如果子代理提出问题**:
- 清晰完整地回答
- 提供额外上下文（如果需要）
- 不要催促他们进入实现

**如果审查者发现问题**:
- 实现者（同一个子代理）修复它们
- 审查者再次审查
- 重复直到批准
- 不要跳过重新审查

**如果子代理失败任务**:
- 分发修复子代理与具体指令
- 不要尝试手动修复（上下文污染）

## 集成

**必需的工作流技能**:
- **using-git-worktrees** - REQUIRED：在执行任何任务之前设置隔离工作空间
- **writing-plans** - 创建此技能执行的计划
- **finishing-a-development-branch** - 所有任务完成后完成开发

**子代理应该使用**:
- **test-driven-development** - 子代理为每个任务遵循 TDD

**替代工作流**:
- **executing-plans** - 用于并行会话而不是同会话执行
