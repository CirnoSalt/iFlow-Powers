# Systematic Debugging（系统性调试）

## 概述

随机修复浪费时间并创建新 bug。快速补丁掩盖潜在问题。

**核心原则：** 在尝试修复之前始终找到根本原因。症状修复就是失败。

**违反这个过程的字面意思就是违反调试的精神。**

## 铁律

```
没有根本原因调查就不能修复
```

如果你没有完成 Phase 1，你就不能提出修复方案。

## 使用时机

用于任何技术问题：
- 测试失败
- 生产环境 Bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别在以下情况使用：**
- 时间紧迫（紧急情况容易让人想猜测）
- "只是快速修复"看起来很明显
- 你已经尝试了多次修复
- 之前的修复不起作用
- 你不完全理解问题

**不要跳过的情况：**
- 问题看起来很简单（简单的 bug 也有根本原因）
- 你在赶时间（匆忙保证返工）
- 经理要求现在修复（系统性方法比反复猜测更快）

## 四个阶段

你必须在进入下一个阶段之前完成每个阶段。

### Phase 1: 根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误消息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完全读取堆栈跟踪
   - 注意行号、文件路径、错误代码

2. **一致地重现**
   - 你能可靠地触发它吗？
   - 确切的步骤是什么？
   - 每次都发生吗？
   - 如果不可重现 → 收集更多数据，不要猜测

3. **检查最近的更改**
   - 什么更改可能导致这个？
   - Git diff、最近的提交
   - 新依赖、配置更改
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复之前，添加诊断工具：**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录退出组件的数据
     - 验证环境/配置传播
     - 检查每一层的状态

   运行一次以收集证据显示在哪里中断
   然后分析证据以识别失败的组件
   然后调查该特定组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== 工作流中可用的机密：==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== 构建脚本中的环境变量：==="
   env | grep IDENTITY || echo "IDENTITY 不在环境中"

   # 第 3 层：签名脚本
   echo "=== 钥匙串状态：==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这揭示：** 哪一层失败（机密 → 工作流 ✓，工作流 → 构建 ✗）

5. **追踪数据流**

   **当错误在调用堆栈深处时：**

   使用 `read_file` 工具读取代码，然后分析数据流：
   - 错误值从哪里开始？
   - 什么用错误值调用了这个？
   - 继续向上追踪直到找到源
   - 在源修复，不是在症状

### Phase 2: 模式分析

**在修复之前找到模式：**

1. **找到工作示例**
   - 在相同代码库中定位类似的正常工作的代码
   - 什么是与损坏的类似的正常工作的？

2. **与参考对比**
   - 如果实现模式，完全读取参考实现
   - 不要略读 - 读取每一行
   - 在应用之前完全理解模式

3. **识别差异**
   - 正常工作和损坏之间有什么不同？
   - 列出每个差异，无论多么小
   - 不要假设"那不可能重要"

4. **理解依赖**
   - 这个需要其他什么组件？
   - 什么设置、配置、环境？
   - 它做出什么假设？

### Phase 3: 假设和测试

**科学方法：**

1. **形成单一假设**
   - 清楚地陈述："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 具体化，不模糊

2. **最小化测试**
   - 进行最小可能的变化来测试假设
   - 一次一个变量
   - 不要一次修复多个事情

3. **在继续之前验证**
   - 有效吗？是的 → Phase 4
   - 无效？形成新假设
   - 不要在顶部添加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助
   - 研究更多

### Phase 4: 实现

**修复根本原因，不是症状：**

1. **创建失败的测试用例**
   - 最简单的可能重现
   - 如果可能，自动化测试
   - 如果没有框架，一次性测试脚本
   - 修复前必须有
   - 使用 `test-driven-development` 技能编写适当的失败测试

2. **实现单一修复**
   - 解决识别的根本原因
   - 一次一个更改
   - 没有"同时在此改进"
   - 没有捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有其他测试被破坏？
   - 问题实际解决了吗？

4. **如果修复不起作用**
   - 停下
   - 计数：你尝试了多少次修复？
   - 如果 < 3：使用新信息返回 Phase 1，重新分析
   - **如果 ≥ 3：停下并质疑架构（下面的步骤 5）**
   - 不要在没有架构讨论的情况下尝试修复 #4

5. **如果 3+ 次修复失败：质疑架构**

   **指示架构问题的模式：**
   - 每次修复在不同的地方揭示新的共享状态/耦合/问题
   - 修复需要"大规模重构"来实现
   - 每次修复在其他地方创建新症状

   **停下并质疑基础：**
   - 这个模式从根本上是否合理？
   - 我们是否"纯粹由于惯性而坚持它"？
   - 我们应该重构架构 vs. 继续修复症状？

   **在尝试更多修复之前与你的合作伙伴讨论**

   这不是失败的假设 - 这是错误的架构。

## 红色标志 - 停下并遵循过程

如果你发现自己正在思考：
- "现在快速修复，稍后调查"
- "只是尝试改变 X 看看是否有效"
- "添加多个更改，运行测试"
- "跳过测试，我会手动验证"
- "可能是 X，让我修复它"
- "我不完全理解但这可能有效"
- "模式说 X 但我会以不同方式适应"
- "这是主要问题：[列出没有调查的修复]"
- 在追踪数据流之前提出解决方案
- **"再来一次修复尝试"（当已经尝试 2+ 次）**
- **每次修复在不同地方揭示新问题**

**所有这些意味着：停下。返回 Phase 1。**

**如果 3+ 次修复失败：** 质疑架构（见 Phase 4.5）

## 你的合作伙伴的信号表明你做错了

**注意这些重定向：**
- "那个没有发生吗？" - 你在没有验证的情况下假设
- "它会向我们显示...？" - 你应该已经添加了证据收集
- "停止猜测" - 你在不理解的情况下提出修复
- "超级思考这个" - 质疑基础，不仅仅是症状
- "我们卡住了？"（沮丧） - 你的方法不起作用

**当你看到这些时：** 停下。返回 Phase 1。

## 常见合理化

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要过程" | 简单的问题也有根本原因。简单问题的过程很快。 |
| "紧急，没有时间过程" | 系统性调试比猜测和检查折腾更快。 |
| "先尝试这个，然后调查" | 第一次修复设置模式。从开始就做正确。 |
| "我会在确认修复工作后写测试" | 未经测试的修复不持久。测试先证明它。 |
| "一次多个修复节省时间" | 无法隔离什么有效。导致新 bug。 |
| "参考太长，我会适应模式" | 部分理解保证 bug。完全读取它。 |
| "我看到问题，让我修复它" | 看到症状 ≠ 理解根本原因。 |
| "再来一次修复尝试"（2+ 次失败后） | 3+ 次失败 = 架构问题。质疑模式，不要再次修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------|----------|
| **1. 根本原因** | 读取错误、重现、检查更改、收集证据 | 理解什么和为什么 |
| **2. 模式** | 找到工作示例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 确认或新假设 |
| **4. 实现** | 创建测试、修复、验证 | Bug 解决，测试通过 |

## 当过程揭示"没有根本原因"

如果系统调查揭示问题真正是环境的、依赖时间的或外部的：

1. 你已完成过程
2. 记录你调查的内容
3. 实现适当的处理（重试、超时、错误消息）
4. 添加监控/日志记录以供未来调查

**但是：** 95% 的"没有根本原因"案例是不完整的调查。

## 在 iFlow CLI 中使用

要在 iFlow CLI 中使用此技能：

```bash
# 调用技能
skill: "systematic-debugging"
```

然后遵循四个阶段：

1. **Phase 1: 根本原因调查**
   - 使用 `read_file` 读取错误消息和代码
   - 使用 `run_shell_command` 重现问题
   - 使用 `run_shell_command` 检查最近的更改

2. **Phase 2: 模式分析**
   - 使用 `read_file` 读取工作示例
   - 对比正常工作和损坏的代码

3. **Phase 3: 假设和测试**
   - 形成假设
   - 使用 `write_file` 或 `replace` 进行最小化更改
   - 使用 `run_shell_command` 验证

4. **Phase 4: 实现**
   - 使用 `test-driven-development` 技能创建失败的测试
   - 使用 `write_file` 或 `replace` 实现修复
   - 使用 `run_shell_command` 验证修复

## 完整示例

```javascript
// Phase 1: 根本原因调查

// 读取错误消息
read_file({
  absolute_path: "/path/to/error.log"
});

// 重现问题
run_shell_command({
  command: "npm test",
  description: "重现错误"
});

// 检查最近的更改
run_shell_command({
  command: "git diff HEAD~1",
  description: "检查最近的更改"
});

// Phase 2: 模式分析

// 读取工作示例
read_file({
  absolute_path: "/path/to/working-example.ts"
});

// 对比差异
read_file({
  absolute_path: "/path/to/broken-code.ts"
});

// Phase 3: 假设和测试

// 形成假设："函数 X 返回 undefined 因为 Y"

// 最小化测试
replace({
  file_path: "/path/to/broken-code.ts",
  instruction: "测试假设",
  old_string: "return result;",
  new_string: "console.log('Result:', result); return result;"
});

run_shell_command({
  command: "npm test",
  description: "验证假设"
});

// Phase 4: 实现

// 使用 TDD 技能创建失败的测试
skill: "test-driven-development"
// 编写失败的测试...

// 实现修复
replace({
  file_path: "/path/to/broken-code.ts",
  instruction: "修复根本原因",
  old_string: "old code",
  new_string: "fixed code"
});

// 验证修复
run_shell_command({
  command: "npm test",
  description: "验证修复"
});
```

## 支持技术

这些技术是系统性调试的一部分：

- **根本原因追踪** - 通过调用堆栈向后追踪 bug 以找到原始触发器
- **深度防御** - 在找到根本原因后在多层添加验证
- **基于条件的等待** - 使用条件轮询替换任意超时

**相关技能：**
- **test-driven-development** - 用于创建失败的测试用例（Phase 4，步骤 1）
- **verification-before-completion** - 在声称成功之前验证修复有效

## 真实影响

从调试会话：
- 系统性方法：15-30 分钟修复
- 随机修复方法：2-3 小时折腾
- 首次修复率：95% vs 40%
- 引入的新 bug：接近零 vs 常见
