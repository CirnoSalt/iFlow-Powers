# Brainstorming 技能使用示例

本文档提供了 Brainstorming 技能的使用示例。

## 基本用法

### 调用技能

在 iFlow CLI 中，你可以通过以下方式调用 Brainstorming 技能：

```
使用 brainstorming 技能来实现这个功能
```

或者：

```
skill: brainstorming
```

### 示例 1：添加新功能

**用户请求**：
```
我想添加一个用户评论功能
```

**AI 响应**：
1. 使用 `read_file` 检查当前项目结构
2. 询问："评论功能应该在哪些页面上可用？"
3. 用户："产品详情页面和博客文章页面"
4. 询问："评论需要哪些功能？"
5. 用户："用户可以发布、编辑和删除自己的评论"
6. 询问："需要评论审核吗？"
7. 用户："是的，管理员需要审核评论"
8. 提出 2-3 种方法：
   - 方法 1：使用现有的评论库（推荐）
   - 方法 2：从头构建自定义评论系统
9. 展示设计：
   - 架构：使用评论库
   - 组件：评论列表、评论表单、评论编辑器
   - 数据流：用户 → 评论 → 数据库 → 审核队列 → 已批准评论
   - 错误处理：验证错误、权限错误
   - 测试：单元测试、集成测试
10. 每部分后获得用户批准
11. 编写设计文档到 `docs/plans/2026-03-05-user-comments-design.md`
12. 调用 writing-plans 技能

### 示例 2：重构现有代码

**用户请求**：
```
我想重构用户认证代码，使其更易于维护
```

**AI 响应**：
1. 使用 `read_file` 检查当前认证代码
2. 询问："当前代码存在哪些问题？"
3. 用户："代码重复，难以测试，缺少错误处理"
4. 询问："重构的目标是什么？"
5. 用户："提高代码可维护性、可测试性和错误处理"
6. 提出 2-3 种方法：
   - 方法 1：提取认证逻辑到单独的服务（推荐）
   - 方法 2：使用现有的认证库
   - 方法 3：完全重写认证系统
7. 展示设计：
   - 架构：认证服务 + 控制器 + 视图
   - 组件：AuthService、AuthController、LoginView、RegisterView
   - 数据流：用户 → 控制器 → 服务 → 数据库
   - 错误处理：统一的错误处理机制
   - 测试：单元测试、集成测试
8. 每部分后获得用户批准
9. 编写设计文档到 `docs/plans/2026-03-05-auth-refactor-design.md`
10. 调用 writing-plans 技能

## 完整对话示例

### 对话 1：简单功能

**用户**：
```
我想在首页添加一个"联系我们"按钮
```

**AI**：
```
我正在使用 brainstorming 技能来创建设计。

让我先检查当前首页的结构...
[使用 read_file 检查首页文件]

这个"联系我们"按钮应该做什么？
```

**用户**：
```
它应该打开一个联系表单模态框
```

**AI**：
```
按钮应该放在页面的什么位置？
```

**用户**：
```
导航栏的右侧，在"登录"按钮旁边
```

**AI**：
```
我提出以下两种方法：

**方法 1（推荐）：使用现有的模态框组件**
- 优点：代码复用，一致的用户体验
- 缺点：需要确认模态框组件支持表单

**方法 2：创建新的联系表单页面**
- 优点：SEO 友好，可以添加更多内容
- 缺点：需要创建新页面，增加导航复杂度

我推荐方法 1，因为它保持了单页应用的一致性。

让我展示设计...

**设计：**

**组件：**
- 使用现有的 Modal 组件
- 创建新的 ContactForm 组件
- 在导航栏添加 ContactButton 组件

**位置：**
- ContactButton 放在导航栏右侧，Login 按钮旁边

**样式：**
- 与其他导航按钮一致
- 悬停效果：轻微放大和颜色变化

**功能：**
- 点击按钮打开模态框
- 模态框包含表单：姓名、电子邮件、消息
- 提交后关闭模态框并显示成功消息
- 表单验证：必填字段、电子邮件格式

这看起来正确吗？
```

**用户**：
```
是的，看起来不错
```

**AI**：
```
好的，让我继续设计...

**数据流：**
1. 用户点击 ContactButton
2. Modal 组件打开
3. ContactForm 组件渲染
4. 用户填写表单并提交
5. 表单数据验证
6. 发送到后端 API
7. 收到响应后关闭模态框
8. 显示成功/错误消息

**错误处理：**
- 网络错误：显示重试选项
- 验证错误：显示字段级错误消息
- 服务器错误：显示通用错误消息

**测试：**
- 单元测试：ContactForm 组件
- 集成测试：模态框打开/关闭
- E2E 测试：完整提交流程

这看起来正确吗？
```

**用户**：
```
是的，继续
```

**AI**：
```
好的，让我编写设计文档...
[使用 write_file 编写设计文档到 docs/plans/2026-03-05-contact-button-design.md]

让我提交设计文档...
[使用 run_shell_command 提交到 git]

设计已完成！现在让我调用 writing-plans 技能来创建实施计划...
[使用 Skill 调用 writing-plans 技能]
```

## 关键要点

1. **一次一个问题**：不要一次问多个问题
2. **提出多种方法**：总是提出 2-3 种方法及其权衡
3. **增量验证**：每部分后获得用户批准
4. **完整文档**：编写完整的设计文档
5. **调用 writing-plans**：最后一步总是调用 writing-plans 技能

## 常见错误

### 错误 1：跳过设计直接写代码

**错误示例**：
```
好的，让我直接添加这个按钮...
[开始编写代码]
```

**正确做法**：
```
让我先探索项目上下文...
提出澄清问题...
展示设计...
获得用户批准...
编写设计文档...
调用 writing-plans 技能
```

### 错误 2：一次问多个问题

**错误示例**：
```
按钮应该放在哪里？它应该做什么？需要什么样式？
```

**正确做法**：
```
按钮应该放在哪里？
[等待用户回答]
它应该做什么？
[等待用户回答]
需要什么样式？
[等待用户回答]
```

### 错误 3：只提出一种方法

**错误示例**：
```
我建议使用现有的模态框组件。
```

**正确做法**：
```
我提出以下两种方法：

**方法 1（推荐）：使用现有的模态框组件**
- 优点：...
- 缺点：...

**方法 2：创建新的联系表单页面**
- 优点：...
- 缺点：...

我推荐方法 1，因为...
```
